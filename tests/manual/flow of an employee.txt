Flow of an employee — rules and lifecycle

1) Registration & Verification
- When a user registers as an employee they are NOT considered a verified employee by default.
- A `verified` flag is required; only the System Admin can mark an employee as verified as the verification in codes after employee submit his verification.
- Unverified employees cannot be invited to companies, cannot belong to an `employments` record, and cannot accept employment invites.

2) Single current employment constraint
- An employee may have at most one active/current employment record (`employments.is_current = true` and `verification_status = 'approved'`).
- Admins or system processes must ensure uniqueness when approving new employments (soft-end existing employment or reject new acceptance).

3) Applications vs Accepting Offers
- Employees may apply to multiple job positions across companies (many applications allowed).
- An employee CANNOT accept an employment offer (hire-invite) from another company while they have an active/current employment.
- To accept a new employment offer, the employee must first end their current employment and have that end request accepted by their current company admin.

4) Ending current employment (Employee-initiated)
- Employee initiates an employment end request (reason optional), which creates an end-request record or sets `employments.end_requested_at`.
- Company Admin receives notification and can either Accept or Reject the end request.
- If Admin Rejects: the employment remains active; the employee cannot accept any external hire-invite until the admin later accepts an end request.
- If Admin Accepts: the employment is marked as ended (soft end / `ended_at` set, `is_current=false`) and the employee is free to accept external hire invites.

5) Hiring / Invite acceptance logic
- Company Admin can send a hire-invite only for applications with status `approved`.
- When admin sends a hire-invite, `job_applications.hire_invite_sent_at` is set and the employee receives an email + notification.
- Employee may only call the accept-hire endpoint if:
  - they are verified, AND
  - either they have no current employment, OR their current employment has an accepted end request.
- On accept-hire: `job_applications.hire_invite_accepted_at` is set; the admin is notified; employment record may be finalized/created depending on the flow.

6) Edge cases & security
- Unverified employees cannot accept invites or be added as `employments`.
- If an employee attempts to accept a hire-invite while still actively employed and the current company has not accepted their end request, API must return 400 with a clear message.
- Only company admin for the target company can send hire-invites for their positions.
- Attempting to accept an invite belonging to another applicant should return 404/403 as appropriate.

7) System invariants to enforce in code / DB
- `employees`: include `is_verified BOOLEAN DEFAULT FALSE`.
- `employments`: enforce at application level that only one row per employee may have `is_current = true` and `deleted_at IS NULL`.
- `job_applications`: include `hire_invite_sent_at`, `hire_invite_accepted_at` (timestamps) and `status` enum (`pending`, `interview`, `approved`, `rejected`).
- All state transitions should be guarded in service layer with clear error messages for unauthorized or invalid transitions.

8) Notifications & UX
- Send notifications for: end-request created, end-request accepted/rejected, hire-invite sent, hire-invite accepted.
- On the UI, show explicit badges: "Offer Pending", "Employed ✓", "End Request Pending", etc., so employees and admins can reason about state.
