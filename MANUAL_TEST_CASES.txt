────────────────────────────────────────────────────────────────────────────────
V2-FIX 6 — EOTM nominee names and vote counts correct  (was: V2-E5, V2-E7) [not all working]
────────────────────────────────────────────────────────────────────────────────
Root cause: eotmService.getEventNominees returned `{ employeeId, name, votes }`
but frontend expected `{ employee_id, full_name, vote_count }`.
Fix: renamed fields in eotmService to match frontend expectations.


[ ] V2-F6-B  Past winners in the "Hall of Fame" section show real employee names
             (not "Unknown") — root cause was `name` vs `employee_name` field.

[ ] V2-F6-C  Each nominee card shows a correct vote count (e.g. "2 votes")
             and the list is sorted highest-votes first.


────────────────────────────────────────────────────────────────────────────────
V2-FIX 7 — Employee EOTM tab shows open events and allows voting  (was: V2-EV1–EV7) [not working]
────────────────────────────────────────────────────────────────────────────────
Root cause: eotmService.getCompanyEvents returned raw eotm_events rows with
`is_active` boolean, but EmployeeDashboard filtered on `ev.status === 'open'`.
With `is_active` present but `status` undefined, no events ever rendered.
Fix: service now maps `status: ev.is_active ? 'open' : 'closed'` on every row.

             →  nominees list expands with real names and vote counts

             vote count increments by 1

[ ] V2-F7-D  Try voting a second time in the same event
             →  an inline error message appears ("You have already voted…")
             →  NO browser alert() dialog

[ ] V2-F7-E  Past winners section still appears below the open events list
             with correct employee names and month/year labels.

[ ] V2-F7-F  Company admin closes the event  →  log back in as bob
             →  open event no longer shows under EOTM tab
             →  closed event winner appears in "Past Winners" section


────────────────────────────────────────────────────────────────────────────────
V2-FIX 8 — Inline errors replace alert() in EOTM actions  (was: V2-E4)
────────────────────────────────────────────────────────────────────────────────
Root cause: handleCreate and handleClose used browser alert() for errors.
Fix: replaced with createError / closeError / voteError inline state.



────────────────────────────────────────────────────────────────────────────────
V2-FIX 9 — Admin verification panel shows real user names  (was: V2-VA after VA7)
────────────────────────────────────────────────────────────────────────────────

`users` (plural table name), so the joined object was `req.users`.
Fix: changed references to `req.users?.full_name` and `req.users?.email`.
